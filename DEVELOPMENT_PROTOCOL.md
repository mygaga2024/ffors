# FFORS 核心代码修改与优化约束协议 (FFORS Development & Optimization Protocol)

## 0. 协议目的
本协议旨在规范 FFORS (Freight Forwarder Ocean Rate System) 项目的 AI 辅助开发过程。由于本项目涉及严密的量化逻辑（如 WoW/MoM 计算）、复杂的外部接口调用（LLM/Webhooks）、容器化部署，**且包含高度敏感的商业报价数据**，任何随意的重构、“过度优化”或疏忽都可能导致牵一发而动全身的系统崩溃或数据泄露。

所有参与本项目开发的 AI Agent 必须严格遵循以下原则，采取**“一步一步来”**的执行策略。

## 1. 核心指导原则：最小干预 (Minimal Intervention)
* **禁止全局重构**：除非得到明确的指令（如：“请重构整个 X 模块”），否则绝对禁止为了“代码更优雅”而修改与当前任务无关的可用代码。
* **增量修改**：所有的功能新增、Bug 修复和性能优化，都必须以**最小侵入性**的方式进行。优先考虑新增函数或类，而不是修改现有核心逻辑。
* **锁定既定逻辑**：凡是涉及核心业务逻辑（如：环比计算公式、PDF 多模态 Prompt、比价雷达权重）的代码，修改前必须二次确认，不得擅自改变业务流。

## 2. 隐私与安全边界 (Security & Data Privacy - 强制执行)
* **数据与代码彻底解耦**：代码库中绝对不得包含任何真实的 Excel/PDF 报价单或隐私数据。必须且只能通过 Docker Volume 挂载外部物理目录（如 `/app/data`）来读取真实数据。
* **代码隐私净化**：提交或生成代码时，必须确保 `.gitignore` 已配置拦截敏感目录和文件（如 `/data/`, `/logs/`, `.env`）。
* **禁止硬编码凭据**：任何数据库密码、API Key、Webhook Token 等机密信息严禁直接写入代码，必须使用 `os.getenv()` 从环境变量读取。
* **禁止打印敏感信息**：严禁在程序终端或日志中直接打印客户名称、真实价格明细等敏感字段。

## 3. “一步一步来”开发规范 (Step-by-Step Execution)
在执行任何复杂指令前，必须按照以下步骤拆解任务，并在每一步完成后等待确认：
1. **先读后写 (Read First)**：在修改任何文件前，必须先读取相关的所有依赖文件、路由定义和数据库模型，理解上下文。
2. **方案确认 (Plan Proposal)**：在正式编写代码前，先输出修改计划（列出涉及的文件名、修改的函数名及思路），并询问是否可以开始写代码。
3. **单点突破 (Single Point)**：每次只修改/新增一个具体功能点。例如，先跑通 Excel 导入，验证无误后，再去写钉钉推送。**严禁一次性提交跨多个模块的大规模代码**。
4. **验证后推进 (Verify & Proceed)**：每完成一个模块的代码生成，需提示进行本地测试。测试通过后，再进入下一个模块的开发。

## 4. 架构与边界隔离约束 (Architecture Boundaries)
FFORS 采用高内聚、低耦合架构，必须严格遵守模块边界：
* **数据接入层 (Ingestion)**：只负责清洗和转换数据，**不得**在此层混入钉钉推送或 AI 行情分析逻辑。
* **核心计算层 (Core Engine)**：FastAPI 的逻辑处理层只负责数据计算（如汇率转换、价格对比），**不得**直接处理文件 I/O 异常，需统一抛出。
* **交互层 (Gateway)**：Webhook 和 API 接口的入参/出参格式一经确认，未经允许**绝对不能修改字段名称或类型**，以免破坏前端/移动端的正常解析。

## 5. 依赖与环境控制 (Environment Control)
* **Docker 环境对齐**：所有代码必须确保能在 Docker 容器内正常运行。涉及到文件路径时，必须使用相对路径或环境变量（如 `os.getenv('DATA_DIR')`），**禁止硬编码本地绝对路径**。
* **严格锁定依赖**：新增第三方 Python 包（如 `pandas`, `fastapi`, `langchain`）时，必须说明理由。未经允许，不得随意更换核心框架或引入重量级库。
* **代理配置感知**：在编写调用国外 API（如 OpenAI/Gemini）的网络请求代码时，必须预留并尊重全局代理设置（`HTTP_PROXY`/`HTTPS_PROXY`）。

## 6. 错误处理与日志 (Error Handling & Logging)
* **静默失败是不可接受的**：核心环节（如数据库写入失败、AI 解析 PDF 失败、钉钉推送超时）必须被 `try-except` 捕获，并记录到隔离的 `/logs` 日志中。
* **避免崩溃**：某个单一报价单的解析失败，绝不能导致整个批量导入任务中断。必须跳过错误项并记录，继续执行后续任务。

## 7. 代码同步指令 (GitHub Sync Protocol)

当需要将本地开发进度同步至远程仓库时，请严格执行以下步骤，确保商业数据隐私与代码基座安全。

### 7.1 隐私审计 (Privacy Audit)
在执行任何 Git 操作前，必须进行自动化扫描与人工确认：
* **忽略名单校验**：检查 `.gitignore` 文件，确保以下路径已被严格拦截：
    * `.env` (包含 API Key、DB 密码)
    * `/data/`, `/uploads/`, `/storage/` (包含真实报价单)
    * `/logs/` (包含运行日志)
    * `__pycache__/` 及其他缓存文件。
* **代码内容扫描**：检索本次修改的代码片段，严禁出现硬编码的 API Key、敏感 IP 地址、以及任何真实的货代供应商或客户名称。

### 7.2 规范化提交 (Standardized Commit)
* **提交信息格式**：编写专业且简洁的 Commit Message。
    * 格式：`<type>: <description>` (例如 `feat: add WoW calculation logic` 或 `fix: database connection timeout`)。
* **内容脱敏**：提交信息中严禁包含任何商业敏感信息。

### 7.3 同步执行程序 (Execution Steps)
请依次执行以下命令，并在每一步完成后记录状态：
1. `git status`：核对变更文件列表，确认无误。
2. `git add .`：将变更加入暂存区。
3. `git commit -m "[你的提交信息]"`：提交到本地仓库。
4. `git push origin main`：推送到远程仓库。

### 7.4 强制约束 (Hard Constraints)
* **禁止强制推送**：严禁执行 `git push --force`，以免覆盖远程仓库的历史记录。
* **冲突处理**：如果遇到远程仓库与本地冲突，必须停止操作并向我报告受影响的文件，严禁擅自处理冲突。
* **结果确认**：同步成功后，请提供最新的 Commit Hash 以供核对。
